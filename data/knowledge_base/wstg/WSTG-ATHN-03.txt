WSTG-ATHN-03: Testing for Weak Lock Out Mechanism (Brute Force Protection)

Summary:
Account lockout mechanisms are used to prevent an attacker from guessing a password through brute force or credential stuffing. Insufficient lockout policies, lack of rate limiting, or bypassable account lockout allows attackers to make unlimited authentication attempts.

Test Objectives:
- Verify that account lockout is enforced after a defined number of failed attempts
- Verify that rate limiting prevents rapid automated authentication attempts
- Identify bypass mechanisms that allow circumventing lockout (IP rotation, username variations)

How to Test:

1. Basic Lockout Testing
   - Attempt to authenticate with an invalid password for a valid account
   - Count how many attempts are permitted before lockout triggers
   - Verify a lockout mechanism is applied (error message change, account disabled)
   - Document the lockout threshold (should be <= 10 attempts for most applications)

2. Lockout Duration Testing
   - After triggering lockout, measure how long until the account can be used again
   - Try to unlock account using "forgot password" flow during lockout
   - Verify lockout is not bypassed by resetting password

3. Rate Limiting Bypass Testing
   - Test from different IP addresses (if lockout is IP-based, it may be bypassed)
   - Vary User-Agent, X-Forwarded-For headers to check if rate limiting uses client-supplied headers
   - Try simultaneous parallel requests to detect race conditions in lockout logic
   - Add whitespace variations to username (e.g., " user@example.com") to test normalization

4. Username Enumeration
   - Compare error messages for valid vs. invalid usernames during lockout
   - Response time differences may also indicate valid username (timing attack)
   - Verify the application returns the same message for both cases

5. CAPTCHA Assessment
   - Verify CAPTCHA is presented after a threshold of failed attempts
   - Test CAPTCHA bypass: reuse old CAPTCHA tokens, test for missing server-side validation

Tools:
   - Burp Suite Intruder: automated credential stuffing simulation
   - Hydra: network login brute forcer
   - Custom scripts with concurrent request support

Expected Secure Behavior (Pass Conditions):
- Account locked or delayed after 5-10 failed attempts
- Lockout applies per-account, not per-IP (to prevent bypass via IP rotation)
- Identical error messages for valid and invalid usernames
- Exponential backoff or progressive delay between attempts
- Alert/notification sent to account owner on lockout
- CAPTCHA or risk-based authentication triggered after threshold
